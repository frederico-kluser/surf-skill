📦 Visão Geral

O surf-skill é um pacote npm (npm i -g surf-skill) que fornece 3 agent skills + 4 CLIs para fazer pesquisa web de alta qualidade a partir de qualquer agente de IA (pi, Claude Code, Copilot, OpenCode).

🎯 Princípio fundamental: O agente NUNCA escolhe o provedor (Tavily, Brave, etc.). A skill decide automaticamente qual API usar, rotaciona chaves, faz fallback, e gerencia custos. O agente só decide o quanto pesquisar — Normal, Parallel, ou Deep.

🎚 Os 3 Modos de Busca

O Mode Router (dentro do SKILL.md) classifica toda pergunta em exatamente um dos 3 modos. É obrigatório rodar o roteador antes de qualquer pesquisa.

Modo Normal — Pesquisa Simples

# Comando único
surf-research-skill search "claude 4.7 release notes" --max 3

# Batch sequencial (vários ângulos, uma chamada)
surf-research-skill search "React 19 new features" "React 19 migration guide" "React 19 performance"

# Modos de profundidade
surf-research-skill search "query" --mode fast    # 1 crédito, ~1-3s
surf-research-skill search "query" --mode normal  # 1 crédito, ~2s
surf-research-skill search "query" --mode slow    # 2 créditos, ~5s (DEFAULT)

Modo Parallel — Fan-out Concorrente

# Com queries inline
surf-research-skill search-parallel \
  "React vs Vue performance benchmarks 2026" \
  "React vs Vue developer experience" \

  --concurrency 6 --json

# Com arquivo JSON de queries (mais controle)
# /tmp/queries.json:
# [

surf-research-skill search-parallel --queries-file /tmp/queries.json --concurrency 8 --json

# No Pi (no-limit), sempre com --no-budget
surf-research-skill search-parallel --queries-file /tmp/queries.json \
  --concurrency 8 --no-budget --json > /tmp/results.json

🔥 Modo Deep — Explicação Completa

O modo Deep é para pesquisas amplas e exaustivas. Ele usa duas estratégias que podem ser combinadas:

Estratégia A: Fan-out Amplo (search-parallel com iteração)

🔁 Iteração em ondas (exclusivo Pi):
  1. Onda 1: search-parallel com 5-10+ sub-questões e 2-4 queries diversas por sub-questão
  2. Avalia o Ledger: alguma sub-questão ficou superficial, contraditória ou sem resposta?
  3. Onda 2: Se sim e < 3 ondas, novo search-parallel focado só nos gaps
  4. Onda 3: (se necessário) última tentativa de preencher lacunas
  5. Hard cap: 3 ondas. O que sobrar é registrado como gap aberto — nunca dropado silenciosamente

Estratégia B: Task API Assíncrona (research-start + research-poll)

Para pesquisas que levam minutos a horas, use o modelo fire-and-forget: 

# 1. Dispara a pesquisa (não bloqueia!)
JOB=$(surf-research-skill research-start "deep dive: WebAssembly GC adoption 2026" \
  --processor pro --confirm-expensive --json | jq -r .data.request_id)

# 2. Poll até completar (cada poll leva <2s e é grátis)
surf-research-skill research-poll "$JOB"

# 3. Quando status mudar para "completed", você tem o relatório completo
# com conteúdo estruturado e fontes citadas

🚀 Como Usar o Deep na Prática (COM SUAS 3 SKILLS)

✅ Você tem tudo que precisa. As 3 skills (surf-research-skill, surf-plan-skill, surf-free-skill) cobrem todos os cenários. O Deep mode NÃO é uma skill separada — é um modo de operação da surf-research-skill.

Passo a Passo para uma Pesquisa Deep

Passo 1: Peça ao agente em linguagem natural

Basta dizer algo como:

"Faça uma pesquisa profunda sobre [TÓPICO]. Quero um relatório completo com
tudo: o que é, como funciona, principais alternativas, benchmarks,
adoção no mercado, limitações, e roadmap futuro. Use Deep mode."

Ou mais simples:

"Deep dive sobre WebAssembly GC — tudo o que existe sobre o assunto."
Passo 2: O agente segue o protocolo do SKILL.md automaticamente

O agente vai:

  1. R0: Detectar que é Pi (no-limit) e que o tópico é amplo → Deep mode
  2. R1: Decompor em sub-questões independentes (ex: 7 sub-questões)
  3. R2: Criar 2-4 queries por sub-questão, cada uma de uma categoria diferente (docs oficiais, comunidade, specs, benchmarks, papers)
  4. R3: Rodar search-parallel com TODAS as queries em paralelo
  5. R4: Preencher o Research Ledger
  6. R5: Avaliar gaps, rodar onda 2 se necessário
  7. R6: Extrair conteúdo completo das melhores URLs
  8. R7: Deduplicar e resolver contradições
  9. R8: Sintetizar com citações inline

Comandos Reais que o Agente Vai Executar (Deep Mode, Onda 1)

# 1. Fan-out amplo com 15-20 queries em paralelo
surf-research-skill search-parallel --queries-file /tmp/surf-queries.json \
  --concurrency 8 --no-budget --json > /tmp/surf-wave1.json

# 2. Extrair os top 2 URLs por sub-questão
surf-research-skill extract --urls-file /tmp/surf-top-urls.json \
  --depth advanced --json > /tmp/surf-extracted.json

# 3. (Opcional) Para pesquisa ultra-profunda, disparar async
JOB=$(surf-research-skill research-start "deep analysis: [TÓPICO ESPECÍFICO]" \
  --processor pro --confirm-expensive --json | jq -r .data.request_id)

# 4. Poll até completar
surf-research-skill research-poll "$JOB"

🌐 O Modo Browser — Layer A vs Layer B

Essa é a parte que faz o surf-skill ser realmente especial. Existem dois caminhos para executar pesquisa, e o sistema escolhe automaticamente:

Layer A — CLI via Bash (preferencial)

surf-research-skill search "query" --max 5
surf-research-skill search-parallel --queries-file q.json --concurrency 8 --no-budget --json
Agente → Bash → surf-research-skill → dispatch.mjs → Tavily API → Parallel API → Brave API

Benefícios: fallback de providers, rotação de keys, cache, controle de custo, retry com jitter.

Layer B — Ferramentas Nativas (fallback quando Bash está bloqueado)

⚠️ Quando usar: Bash bloqueado (ex: modo plan/approval), CLI não instalada, ou harness restringe ferramentas Bash.

Quando o Bash está indisponível, o agente usa as ferramentas nativas do harness. No pi, essas ferramentas são web_search e fetch_content:

Agente → web_search → Navegador/APIs → Resultados → fetch_content → Conteúdo completo

Como o Layer B Funciona na Prática

O agente segue exatamente o mesmo protocolo do Layer A, mas usa ferramentas diferentes: 

# Layer A (CLI Bash):
surf-research-skill search-parallel "q1" "q2" "q3" --concurrency 6 --json
surf-research-skill extract url1 url2 url3 --depth advanced --json

# Layer B (equivalente com ferramentas nativas do pi):
# 1. Múltiplas queries em UMA rodada (executam em paralelo!)
web_search({ queries: ["q1", "q2", "q3"] })

# 2. Extrair conteúdo completo dos top hits
fetch_content({ urls: ["url1", "url2", "url3"] })

O segredo: o SKILL.md instrui o agente a emitir TODAS as chamadas WebSearch em uma única rodada ("one turn"), o que faz as ferramentas nativas executarem concorrentemente — igual ao search-parallel da CLI.

O Research Ledger — A Ponte Entre as Camadas

WaveSub-QQuery IDProviderStatusTop SourceKey Fact
1sq1sq1-docstavily✅ OKWebAssembly spec — webassembly.github.ioGC é um post-MVP proposal...

🔄 Sistema de Fallback e Retry

Hierarquia de Fallback por Operação

OperaçãoCadeia de Fallback
searchtavily → parallel → brave
extracttavily → parallel
crawltavily only
maptavily only
research-startparallel → tavily

O last_ok_provider é promovido para a frente da cadeia — se o Tavily funcionou na última chamada, ele lidera a próxima.

Ciclo de Retry por Provider/Key

Para cada provider na cadeia:
  Para cada key (round-robin, pula queimadas/cooldown):
    Para cada tentativa (máx 3):
      
      ├─ 4xx (bad input) → ABORTA (não adianta retentar)
      ├─ 429 (rate limit) → backoff com jitter: 1.5s² → 3.3s² → 8s²
      │   └─ 3 falhas → key entra em COOLDOWN 60s (persistido)
      ├─ 401/403 (auth)   → key marcada como BURNED (auto-reset mensal)
      ├─ 5xx (server)     → backoff com jitter, 3 tentativas
      │   └─ 3 falhas consecutivas → key BURNED
      └─ Network error    → backoff com metade do jitter, 3 tentativas

Backoff Exponencial com Jitter Igual

function backoff(attempt) {
  // Teto: 1.5s * (1+1)² = 6s, máximo 8s
  const capped = Math.min(1500 * (attempt + 1) ** 2, 8000);
  // Metade fixo + metade aleatório — evita tempestades de retry
  return Math.round(capped / 2 + Math.random() * (capped / 2));
}

// Tentativa 0: ~750ms  + random(0..750)
// Tentativa 1: ~3000ms + random(0..3000)
// Tentativa 2: ~4000ms + random(0..4000)

Auto-Budget: Detectando Timeout do Harness

Antes de cada chamada HTTP, o dispatch verifica se ainda há tempo dentro do budget do harness: 

const elapsed = Date.now() - startTs;
const remaining = harnessBudget - elapsed - 2000; // 2s de margem
if (remaining <= 1000) {
  throw Error('LikelyAgentTimeout: esta chamada excederia o timeout do agente. ' +
    'Rode "surf-research-skill project-config" ou use research-start + research-poll.');
}

Isso evita que o processo morra com SIGTERM silencioso — em vez disso, produz um erro acionável com a solução.

🔐 Segurança das Chaves

~/.config/surf/keys.json    ← chmod 600 (só o dono lê)
~/.cache/surf/audit.log     ← registra provider + índice, NUNCA a chave

Propriedades de segurança:

  • Nunca lê de env vars em runtime (só na migração inicial)
  • Keys mascaradas nos logs: tvly-…ab12
  • Atomic writes: escreve em .tmp, renomeia (sem arquivo corrompido)
  • Lockfile (.keys.lock): previne race conditions
  • Auto-reset mensal de keys queimadas
  • Validação live: keys add testa contra a API antes de salvar
